# Contributing to ForgeOS

ForgeOS is a methodology, not a product. Contributions that clarify the model, add useful templates, or share real-world patterns are welcome.

## What We're Looking For

### High Value
- **Role refinements** - Better prompts for Architect, UX, QA, Process roles
- **Domain-specific roles** - Security Engineer, Data Engineer, DevOps, etc.
- **Build Prompt patterns** - Templates that work well for specific scenarios
- **Canonical file templates** - Better SPEC.md, ARCH.md, FLOWS.md starters
- **Real-world examples** - Anonymized examples from production use

### Also Welcome
- Typo fixes and clarity improvements
- Translations
- Tool-specific adaptations (though keep it tool-agnostic where possible)

### Not Looking For
- Tool-specific integrations that lock in vendors
- Automation that removes human decision points
- Complexity for its own sake

## How to Contribute

1. **Open an issue first** for significant changes
2. **Fork and branch** from `main`
3. **Keep changes focused** - one concept per PR
4. **Test with real work** if possible - does this actually help?
5. **Submit PR** with clear description of what and why

## Style Guidelines

### For Documentation
- Clear over clever
- Concrete examples over abstract explanations
- Actionable guidance ("do this") over theory ("you should consider")
- Tables and checklists where they aid clarity

### For Templates
- Minimal viable structure - don't over-engineer
- Comments explaining the "why"
- Placeholders that are obviously placeholders (e.g., `[Your Feature Name]`)

## Questions?

Open an issue. We'd rather answer questions than have people guess.
